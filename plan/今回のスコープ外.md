# All-in-one Music Structure Analyzer 拡張設計書
## 目的
既存の Music Structure Analyzer に対して、再生中にコード進行を表示する機能を追加する。
本機能は、既存の構造解析結果（beats / downbeats / segments / bpm）を時間基準として利用し、madmom 等で推定したコード列をタイムライン上に重ねて表示する。

## 1. 追加する機能の責務
本機能の責務は次の 3 点に限定する。

1. 音源からコード候補を推定する。
2. 推定結果を既存の beat / downbeat 基準に合わせて整形する。
3. 再生位置に応じて、現在のコードと進行を UI に表示する。

既存の構造解析機能（セクション分割、テンポ推定、beat 検出、downbeat 検出）は変更しない。
本機能はそれらの上に載る「コードレイヤー」として実装する。

## 2. 前提データ
既に以下のデータが取得できているものとする。

- path
- bpm
- beats
- downbeats
- beat_positions
- segments
- tempo_candidates

このうち、本機能で最も重要なのは beats と downbeats である。
コード表示の時間合わせは、この 2 つを基準に行う。

## 3. 全体構成
処理は次の 4 層で構成する。

### 3.1 解析層
音源からコード推定を行い、時刻付きの chord segment を得る。

例:
- start: 0.27
- end: 2.12
- label: C

### 3.2 整形層
推定結果の短すぎる揺れを抑え、ラベル正規化を行う。
必要に応じて Bm7 → Bm、G6 → G のような表示簡略化を行う。

### 3.3 スナップ層
コード区間の開始・終了時刻を、beat / downbeat に吸着させる。
このとき、前後の区間との整合性を見て、単純な最近傍ではなく安定化処理を入れる。

### 3.4 表示層
再生中の現在時刻に対して、現在のコードを 1 個表示する。
必要に応じて、進行中の複数コードをタイムライン上にブロック表示する。

## 4. データ構造
コード関連データは、内部用と表示用を分ける。

### 4.1 内部用 chord segment
    {
      "start": 0.27,
      "end": 2.12,
      "label_raw": "Bm7",
      "label_norm": "Bm",
      "confidence": 0.83
    }

### 4.2 表示用 chord segment
    {
      "start": 0.27,
      "end": 2.12,
      "label": "Bm"
    }

### 4.3 表示状態
    {
      "current_time": 123.45,
      "current_chord": "Em",
      "current_segment_index": 17
    }

## 5. 処理フロー
処理の流れは次の通りとする。

1. 音源を読み込む。
2. 既存の構造解析結果を読み込む。
3. コード推定を行う。
4. コードラベルを正規化する。
5. chord segment を beat / downbeat にスナップする。
6. 短すぎる区間を統合する。
7. 再生中の時刻に応じて表示する。

## 6. コード推定
madmom を使う場合、コード推定は区間ラベル列として取得する。
ただし、モデルの生出力は揺れやすいため、そのまま表示しない。

実装上は次の方針を取る。

- 推定器が出した raw chord segment を受け取る
- 短いノイズ区間を除去する
- 表示用に必要ならコード品質を簡略化する
- 再生同期しやすいように beat 基準へ整える

## 7. コード正規化
表示の安定性を優先し、必要に応じてコードを簡略化する。

### 7.1 正規化ルール例
- Bm7 → Bm
- G6 → G
- Cmaj7 → C
- Am7 → Am

### 7.2 正規化方針
正規化は「内部推定を壊さない」ことを前提に、表示専用ラベルとして使う。
研究用途では raw label も保持する。

### 7.3 方針の使い分け
- 研究ログ: raw label を保存
- UI 表示: norm label を優先
- 進行比較: norm label を使用
- 精密解析: raw label を参照

## 8. Beat スナップ（前後考慮）
この機能の中心は、コード区間の境界を beat / downbeat に揃える処理である。

### 8.1 基本方針
単純に「最も近い beat に丸める」だけでは、境界が前後に揺れやすい。
そのため、以下の 3 段階でスナップする。

1. 候補 beat を前後から探す
2. 前後の区間との整合性を評価する
3. 最終的に最も安定する境界を採用する

### 8.2 優先順位
境界候補の優先順位は次の通りとする。

1. downbeat
2. beat
3. 元の時刻を維持

downbeat は拍頭として視覚的な安定性が高いため、近傍にある場合は優先する。

### 8.3 前後考慮付き境界スナップ
各 chord segment の start と end を、それぞれ独立に最近傍へ丸めると、隣接区間が食い込んだり、空白が生じたりする。
これを避けるため、前後のセグメントと合わせて境界を決める。

#### start 境界
対象区間の開始時刻 t に対し、前後の候補 beat を取る。

- left = t 以下の最も近い beat / downbeat
- right = t 以上の最も近い beat / downbeat

評価の考え方:
- left に寄せると前区間と自然につながるか
- right に寄せると区間が短くなりすぎないか
- downbeat が近いならどちらを優先するか

#### end 境界
対象区間の終了時刻 t に対しても同様に、前後の候補を比較する。

- left = t 以下の最も近い beat / downbeat
- right = t 以上の最も近い beat / downbeat

評価の考え方:
- 次区間との重なりを避ける
- 1拍未満の短さにならないようにする
- downbeat に落ちるなら優先する

### 8.4 スナップ判定ロジック
境界候補を複数評価し、最終的に次の点数で選ぶ。

- 元の時刻からの距離が小さい
- 区間長が極端に短くならない
- 前後区間と順序が崩れない
- downbeat に一致する場合は加点

### 8.5 実装イメージ
    def choose_boundary(t, beats, downbeats, prev_boundary=None, next_boundary=None, side="start"):
        candidates = collect_candidates_around(t, beats, downbeats)

        best = None
        best_score = float("inf")

        for c in candidates:
            score = abs(c - t)

            if is_downbeat(c, downbeats):
                score -= DOWNBEAT_BONUS

            if prev_boundary is not None and side == "start":
                if c < prev_boundary:
                    score += ORDER_PENALTY

            if next_boundary is not None and side == "end":
                if c > next_boundary:
                    score += ORDER_PENALTY

            if violates_min_duration(c, prev_boundary, next_boundary, side):
                score += SHORT_SEGMENT_PENALTY

            if score < best_score:
                best_score = score
                best = c

        return best if best is not None else t

### 8.6 スナップ後の後処理
スナップ後に次を実施する。

- duration が短すぎる区間を隣接区間と統合する
- 同一ラベルの連続区間を結合する
- 1 拍未満の点滅区間を除外する

### 8.7 推奨パラメータ
- MIN_CHORD_DURATION = 0.30 秒
- SNAP_WINDOW = 0.35 秒
- DOWNBEAT_BONUS は beat より少し大きめ
- SHORT_SEGMENT_PENALTY は強め

数値は音源と UI 表示の応答性を見ながら調整する。

## 9. 表示ロジック
表示は 2 系統に分ける。

### 9.1 現在コード表示
再生中の現在時刻 t に対して、t を含む chord segment の label を表示する。

    def get_current_chord(t, chord_segments):
        for seg in chord_segments:
            if seg["start"] <= t < seg["end"]:
                return seg["label"]
        return None

### 9.2 タイムライン表示
全 chord segment を横並びのブロックとして描画する。
既存の segments 表示の上または下に、追加レイヤーとして重ねる。

表示内容:
- コード名
- 開始位置
- 終了位置
- 必要なら confidence

## 10. 既存データとの役割分担
既存の構造解析データはそのまま活用する。

### 10.1 beats
- 再生カーソルの基準
- 表示の細かな同期
- スナップ候補の生成

### 10.2 downbeats
- 小節頭の強い基準
- コード境界の安定化
- セクション頭の可視化

### 10.3 segments
- intro / verse / chorus などの大区間表示
- コードレイヤーを載せる土台
- 解析結果全体の文脈表示

### 10.4 beat_positions
- 拍番号表示
- 1 拍目の強調
- UI 上での拍目ラベルの補助

## 11. 再生中の更新方式
デモアプリでは、再生中に 30fps 程度の UI 更新を行い、コード表示は current_time の更新に合わせて行う。

更新の基本方針:
- 再生時刻を定期取得する
- current_time から current_chord を引く
- UI の現在表示だけ更新する
- コード変更がなければ再描画を抑える

## 12. 画面構成
既存 UI に対して、次の追加を行う。

### 12.1 上段
- 波形、スペクトル、ピッチ等の既存表示

### 12.2 中段
- 既存のセクション帯
- その上または下にコード帯を追加

### 12.3 下段
- 現在コード
- 現在セクション
- 拍位置
- 再生位置

## 13. 研究用途としてのログ保存
研究用途のため、表示だけでなくログも保存する。

保存推奨項目:
- raw chord segments
- normalized chord segments
- snapped chord segments
- current time と current chord の遷移履歴
- 変更理由のメタ情報
- パラメータ設定値

## 14. 例外処理
次のケースを想定する。

### 14.1 コード推定が不安定な場合
- 短時間で頻繁に変化するラベルを抑制する
- confidence が低い区間は暫定表示にする

### 14.2 beat 情報が欠損する場合
- downbeat を優先する
- beat がない区間は raw time ベースで表示する

### 14.3 セグメントが極端に短い場合
- 隣接区間へ統合する
- 表示対象から外す

### 14.4 終端付近
- 最終 chord segment は音源長に合わせて end を切る
- end の外側は表示しない

## 15. 実装順
実装は次の順序がよい。

1. chord segment を JSON で追加保存する
2. current_time から current_chord を取得する
3. タイムラインに chord 帯を描く
4. beat / downbeat に snap する
5. 前後考慮の安定化を入れる
6. 正規化ルールを追加する
7. ログ保存を入れる

## 16. 最小実装の完成条件
次が満たされれば、デモとして成立する。

- 再生中に current chord が表示される
- chord 帯がタイムラインに重なる
- beat / downbeat に大きなズレがない
- 短い誤検出が目立たない
- 既存の structure analyzer の表示を壊さない

## 18. 補足
この設計では、コード認識そのものを UI に直結しない。
まず推定、次に整形、最後に表示という順で分離することで、再生中表示の安定性と研究用途の再現性を両立する。
