# 小節単位キー検出モジュール - 要件定義書

## 1. はじめに

### 1.1 目的
本ドキュメントは、既存の「all-in-one」プロジェクトに小節単位のキー（調性）検出機能を追加するための要件を定義する。

### 1.2 背景
既存の [`allin1`](src/allin1/) プロジェクトは、音声ファイルから以下の情報を高精度に抽出できる：
- ビート位置（`beats`）
- ダウンビート位置（`downbeats`）→ **小節の頭を特定可能**
- セグメント構造（`segments`：intro, verse, chorus など）
- BPM 推定値

しかし、キー（調性）情報は含まれていない。本機能は、このギャップを埋めることを目的とする。

### 1.3 スコープ

#### 範囲内（In-Scope）
- Demucs で分離された音源（vocals, drums, bass, other）のうち、**ボーカル・ドラムを除いた音源**（bass + other）を使用
- librosa を用いた和音検出
- 小節単位でのキー推定
- セグメントごとのキー集約（intro/verse/chorus などの機能的セグメントごとに代表キーを出力）

#### 範囲外（Out-of-Scope）
- モデルの再学習（既存の Harmonix モデルを流用）
- リアルタイム処理（バッチ処理を想定）
- キー変化の自動検出閾値の最適化（デフォルト値を設定し、後で調整可能とする）

## 2. 機能要件

### FR-1: 音源分離と再ミックス

| ID | 要件名 | 説明 | 優先度 |
|----|--------|------|--------|
| FR-1.1 | Demucs 出力の読み込み | `demix/htdemucs/{track_name}/` ディレクトリから `bass.wav`, `other.wav` を読み込む | 必須 |
| FR-1.2 | 再ミックス処理 | `bass.wav` と `other.wav` を加算混合し、キー分析用音源を生成する | 必須 |
| FR-1.3 | サンプリングレート統一 | 入力音源のサンプリングレートを検出アルゴリズムに合わせる（デフォルト：44100Hz） | 必須 |

### FR-2: 和音検出

| ID | 要件名 | 説明 | 優先度 |
|----|--------|------|--------|
| FR-2.1 | チェルニーシーケンス検出 | librosa または madmom を用いて、時間軸に沿った和音シーケンスを検出する | 必須 |
| FR-2.2 | 和音ラベル形式 | 標準的な和音表記（例：`C`, `Am`, `F#m7b5`, `G7`）を使用する | 必須 |
| FR-2.3 | 検出間隔 | 1 フレームあたり 10ms～50ms の解像度で和音を推定する | 必須 |

### FR-3: キー推定

| ID | 要件名 | 説明 | 優先度 |
|----|--------|------|--------|
| FR-3.1 | 小節単位のキー推定 | ダウンビートで区切られた各小節について、和音シーケンスからキーを推定する | 必須 |
| FR-3.2 | セグメント集約 | 同じセグメント（例：verse, chorus）に属する小節のキーを集約し、代表キーを決定する | 必須 |
| FR-3.3 | キー変化検出 | セグメント間でキーが変更されている場合、そのことを明示的に記録する | 必須 |

### FR-4: 出力形式

| ID | 要件名 | 説明 | 優先度 |
|----|--------|------|--------|
| FR-4.1 | JSON 出力 | 分析結果を JSON 形式で保存する | 必須 |
| FR-4.2 | AnalysisResult 拡張 | [`AnalysisResult`](src/allin1/typings.py:32) データクラスにキー情報を追加する | 必須 |

## 6. 用語定義

| 用語 | 定義 |
|------|------|
| **ダウンビート** | 小節の頭（1 拍目）に位置する強いビート |
| **セグメント** | 曲の構造的な区切り（intro, verse, chorus など） |
| **キー（調性）** | 楽曲の基盤となる音階（例：C 長調、A 短調） |
| **和音** | 複数の音が同時に鳴った状態（例：C major = C-E-G） |
